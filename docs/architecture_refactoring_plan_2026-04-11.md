# Архитектурный анализ и план рефакторинга (2026-04-11)

## 1) Текущее состояние архитектуры
- Монолит FastAPI-приложения с несколькими роутерами (`auth`, `pages`, `complaint`, `exam_import`, `admin`).
- Вход в приложение и сборка зависимостей сосредоточены в `app.py`: middleware, health checks, инициализация сторов, task registry.
- Слой хранения исторически был смешанным, но целевой и текущий runtime-путь теперь закреплен за PostgreSQL.
- Shared-домен (`shared/`) используется как ядро для построения BBCode/валидации, но адаптация payload->domain выполняется в web-сервисах.
- Шаблоны Jinja2 + vanilla JS формируют SSR/MPA-подход.

## 2) Главные архитектурные риски
- Высокая связанность web-слоя и инфраструктурного слоя (инициализация, middleware, health, background registry в одном файле).
- Историческое смешение нескольких storage-подходов в одном классе хранения усложняло эволюцию схемы и повышало риск регрессий.
- Потоковые background задачи импортов на `threading` без явной очереди/воркера и с хранением статуса в БД.
- Глобальные синглтоны backend/секретов и env-зависимое поведение ухудшают предсказуемость тестов.

## 3) Тонкие/хрупкие места
- Логика прав администратора завязана на username и env fallback.
- CSRF-защита реализована через origin-check middleware, но без CSRF token механики для cookie-auth сценариев.
- Часть отказоустойчивости реализована fallback-режимами (например rate limiter), что может маскировать проблемы инфраструктуры.
- Преобразования и валидации разбросаны между route/service/shared слоями.

## 4) Проблемы по слоям
- API-слой: маршруты содержат бизнес-решения (валидация server payload, логирование метрик по событию).
- Service-слой: часть сервисов зависит от FastAPI `HTTPException` вместо доменных ошибок.
- Storage-слой: SQL/миграции/маппинг в одном месте; много conditionals по backend.
- Config/security: критичные настройки читаются из env и кешируются глобально, что создает скрытые состояния.

## 5) Что критично исправить
1. Ввести четкие контракты между Route -> UseCase(Service) -> Repository.
2. Закрепить PostgreSQL-only реализации репозиториев и дочистить legacy-хвосты в storage/API слое.
3. Убрать бизнес-ошибки из HTTP-слоя: доменные исключения маппить в роутере.
4. Централизовать политику аутентификации/авторизации и сделать явный policy layer.
5. Зафиксировать транзакционные границы и единый стиль работы с БД.

## 6) Что можно отложить
- Косметическая реорганизация фронтенд-ассетов.
- Глубокая переработка шаблонов, если нет UX требований.
- Оптимизации перфоманса до стабилизации контрактов и тестов.

## 7) План P0/P1/P2/P3
- P0 (стабилизация): тестовый safety-net, инвентаризация маршрутов/сервисов/хранилищ, фиксация текущих контрактов.
- P1 (разделение слоев): выделение use-case сервисов и domain errors; маршруты делают только I/O и mapping.
- P2 (storage рефакторинг): backend-specific repositories + общий интерфейс; миграция вызовов store.
- P3 (hardening): observability, cleanup middleware composition, улучшение security (csrf strategy), техдолг по фронтенду.

## 8) Что делать первым
- Построить архитектурную карту (таблица endpoint -> service -> repository -> backend).
- Вынести 1 вертикаль как шаблон (например complaint generate flow), провести через новый контрактный слой.

## 9) Что страховать тестами до рефакторинга
- Auth flows (register/verify/login/reset/change password).
- Complaint generate/rehab generate и валидации payload.
- Role-based доступ (admin/tester/gka) и page/api ограничения.
- DB-совместимость: postgres runtime path.
- Rate-limit поведение и fallback mode.
- Background exam import: lifecycle task queued/running/completed/failed.
