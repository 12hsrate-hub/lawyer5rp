# Варианты будущего дизайна и фронтенд-стека (эффективность по затратам)

## Текущая точка проекта

На текущий момент веб-часть уже работает как FastAPI + серверные HTML-шаблоны + отдельные JS/CSS-ресурсы.
Это означает, что базовая SSR-архитектура уже развёрнута и поддерживается командой.

## Что считать «эффективным по затратам»

Для этого проекта эффективность = баланс из 4 факторов:

1. **Скорость вывода улучшений в UI** (time-to-market).
2. **Стоимость разработки и поддержки** (часы команды, сложность кода).
3. **Риск миграции** (вероятность сломать существующие пользовательские сценарии).
4. **Производительность и SEO** (быстрая загрузка страниц, индексируемость).

## Вариант A — Эволюция текущего SSR (рекомендуется как базовый)

### Стек

- FastAPI + Jinja2 (оставляем).
- Текущий vanilla JS, но структурируем по модулям/страницам.
- Токенизированная дизайн-система на CSS-переменных.

### Когда подходит

- Когда нужен быстрый редизайн без полной смены технологической платформы.
- Когда важен бюджет и минимизация рисков.

### Плюсы

- Минимальная стоимость внедрения (используется текущая архитектура).
- Почти нет затрат на инфраструктурную миграцию.
- Хорошо для форм, кабинетов, админок и юридических сценариев с server-first логикой.

### Минусы

- При росте сложных интерактивных экранов JS-слой может усложняться.
- Меньше готовых UI-экосистем, чем у React/Vue.

### Оценка затрат (относительно)

- Вход: **низкий**.
- Поддержка: **низкая/средняя**.
- Риск: **низкий**.

## Вариант B — FastAPI + HTMX (плавный шаг к более «живому» UI)

### Стек

- FastAPI + Jinja2.
- HTMX для частичных обновлений HTML без SPA.
- Минимальный JS для специфичных сценариев.

### Когда подходит

- Когда нужны более «динамичные» формы/таблицы/виджеты,
  но без стоимости полноценного SPA.

### Плюсы

- Очень хорошее соотношение «функциональность/затраты».
- SSR-подход сохраняется, команда не теряет текущие компетенции.
- Упрощает сложные интерактивные участки (пагинация, фильтры, inline-редактирование).

### Минусы

- Потребует внедрения новых UI-паттернов и дисциплины по endpoint-структуре.
- Не заменяет полностью потребность в JS при сложной клиентской логике.

### Оценка затрат (относительно)

- Вход: **низкий/средний**.
- Поддержка: **средняя**.
- Риск: **низкий/средний**.

## Вариант C — SPA на React/Vue + API (максимальная гибкость, выше стоимость)

### Стек

- Отдельный фронтенд (React/Vue + Vite/Next/Nuxt).
- FastAPI как чистый API backend.

### Когда подходит

- Когда продукту нужен сложный интерактивный интерфейс масштаба SaaS.
- Когда есть стабильная фронтенд-команда и бюджет на поддержку двух приложений.

### Плюсы

- Максимальная гибкость UI/UX.
- Широкая экосистема компонентов и библиотек.

### Минусы

- Самая высокая стоимость владения.
- Двойная сборка/деплой/наблюдаемость (frontend + backend).
- Существенные миграционные риски.

### Оценка затрат (относительно)

- Вход: **высокий**.
- Поддержка: **высокая**.
- Риск: **средний/высокий**.

## Рекомендация для lawyer5rp

Оптимальный путь по цене/скорости/риску:

1. **Этап 1 (сейчас): Вариант A** — сделать редизайн в текущей SSR-архитектуре.
2. **Этап 2 (точечно): добавить элементы Варианта B (HTMX)** для самых «живых» экранов.
3. **Этап 3 (опционально): рассматривать SPA только при доказанной бизнес-необходимости**.

Иными словами: **не делать «big bang»-миграцию**.

## Практический план (6–8 недель)

1. **Недели 1–2:** UI-аудит и дизайн-токены.
2. **Недели 2–4:** обновление базового layout, типографики, форм, таблиц, кнопок.
3. **Недели 4–6:** редизайн 2 ключевых страниц (например, `complaint`, `profile`) с UX-метриками.
4. **Недели 6–8:** точечное внедрение HTMX в узких местах (где выигрыш очевиден).

## Куда вносить изменения в первую очередь (чтобы визуально стало лучше быстро)

Если цель — заметный результат без больших рисков, правки лучше делать слоями,
в таком порядке:

1. **Базовый layout и сетка**
   - Единый контейнер контента (max-width, отступы по брейкпоинтам).
   - Единая вертикальная ритмика между секциями.
   - Предсказуемая адаптивность на 360/768/1024/1440 px.
2. **Типографика и цветовые токены**
   - Шкала размеров заголовков/текста.
   - Контрастные пары текста/фона (особенно для вторичного текста).
   - Состояния ссылок: default/hover/active/visited.
3. **UI-компоненты высокого эффекта**
   - Кнопки (primary/secondary/ghost + disabled/loading).
   - Поля форм (focus/error/success/help text).
   - Карточки, таблицы, алерты и пустые состояния.
4. **Ключевые пользовательские сценарии**
   - Экран `complaint`: пошаговость, визуальный прогресс, понятные ошибки.
   - Экран `profile`: читаемость блоков, быстрые действия, статусные бейджи.
5. **Микро-взаимодействия**
   - Hover/focus-реакции до 150–200ms.
   - Skeleton/loading для участков с ожиданием ответа сервера.
   - Уведомления об успехе/ошибке без перезагрузки (где уместно через HTMX).

> Практическое правило: сначала стабилизируйте визуальную основу (layout + токены +
> базовые компоненты), и только потом делайте «украшения» на отдельных страницах.

## Конкретный промт для точечных улучшений (можно копировать в Codex/LLM)

Используйте этот шаблон для каждой страницы/компонента:

```text
Ты senior frontend/UX инженер в проекте на FastAPI + Jinja2 + vanilla JS.
Нужно предложить ТОЛЬКО точечные улучшения визуализации без миграции на SPA.

Контекст:
- Проект: lawyer5rp
- Цель: улучшить читаемость, визуальную иерархию и конверсию в ключевом сценарии.
- Ограничения: минимум риска, без полного переписывания фронтенда.
- Текущий экран: <вставь имя страницы, например complaint/profile>
- Текущие проблемы: <список 3–7 наблюдений>

Сделай результат в формате:
1) UX-аудит (топ-10 проблем по приоритету: Critical/High/Medium/Low).
2) Предложение quick wins на 1–2 дня и improvements на 1–2 недели.
3) Конкретные правки по слоям:
   - layout/сетка
   - типографика
   - цвета/контраст
   - формы и кнопки
   - таблицы/карточки/пустые состояния
   - mobile-адаптация
4) Готовые CSS-токены (variables) и примеры классов (BEM или utility-подход).
5) Минимальные изменения в шаблонах Jinja2 (какие блоки/элементы тронуть).
6) Критерии приёмки (чеклист) + метрики до/после (LCP, time-to-task, error rate).

Важно:
- Не предлагай полную смену стека.
- Для каждого пункта укажи ожидаемый эффект и трудозатраты (S/M/L).
- Покажи примеры кода небольшими фрагментами (без огромных простыней).
```

## Расширенное внедрение: чёткий план на 12 недель

Ниже — более детальный план внедрения с конкретными артефактами, ролями и
критериями готовности. Его можно использовать как рабочий трекер в спринтах.

### Цели внедрения (что считаем успехом)

1. **Визуальная консистентность:** единый стиль на ключевых экранах (`home`,
   `complaint`, `profile`, auth-формы).
2. **Скорость сценария:** сокращение времени прохождения ключевого пути
   «заполнить и отправить жалобу».
3. **Снижение ошибок:** меньше клиентских и серверных ошибок валидации.
4. **Прозрачная поддержка:** изменения UI должны вноситься быстрее и предсказуемее
   (через дизайн-токены, стандарты компонентов и чеклисты).

### Роли и зоны ответственности

- **Product/Owner:** приоритизирует сценарии и KPI по бизнес-эффекту.
- **Frontend/Template инженер:** вносит изменения в Jinja2/CSS/JS и UI-компоненты.
- **Backend инженер:** поддерживает endpoint-ы для HTMX/частичных обновлений.
- **QA:** сценарная проверка (desktop/mobile), регресс и доступность.
- **Аналитика:** фиксирует метрики до/после и готовит weekly-отчёт.

### Бэклог внедрения (по приоритету)

#### P0 (обязательно)

1. Дизайн-токены: цвета, типографика, spacing, радиусы, тени, z-index.
2. Базовый layout: контейнеры, grid, адаптивные отступы.
3. Библиотека базовых компонентов:
   - Button, Input, Select, Textarea, Checkbox, Radio.
   - Alert, Badge, Card, Table, EmptyState.
4. Единые состояния:
   - hover/focus/disabled/loading/error/success.
5. 2 ключевых экрана:
   - `complaint` и `profile` (сквозная визуальная консистентность).

#### P1 (высокий приоритет)

1. HTMX-точки для «узких мест» (фильтры, списки, частичные обновления секций).
2. Skeleton/loading + inline-уведомления об успехе/ошибке.
3. Улучшение мобильной версии (360–430 px): типографика, кнопки, формы, таблицы.
4. Паттерны пустых/ошибочных/пограничных состояний.

#### P2 (опционально)

1. Тёмная тема (если есть бизнес-запрос).
2. Расширенные motion-гайды (без ущерба производительности).
3. Storybook-подобная документация компонентов (или markdown-каталог).

### План-график по неделям

#### Фаза 0 — Подготовка (Неделя 1)

- Зафиксировать baseline:
  - список экранов в scope;
  - текущие проблемы UX/UI;
  - метрики LCP, time-to-task, error rate.
- Создать доску внедрения (эпики: tokens/layout/components/screens/qa).
- Утвердить Definition of Done для UI-задач.

**Артефакты:** baseline-таблица, backlog P0/P1/P2, DoD-чеклист.

#### Фаза 1 — Основа дизайн-системы (Недели 2–3)

- Ввести CSS variables (токены) и слой базовых utility-классов.
- Обновить базовый layout и брейкпоинты.
- Внедрить базовые типографические стили.

**Артефакты:** token-map, обновлённый base stylesheet, гайд по использованию.

#### Фаза 2 — Компоненты (Недели 4–5)

- Перевести формы/кнопки/алерты/карточки/таблицы на единые компоненты.
- Нормализовать состояния и поведение ошибок валидации.
- Добавить единые шаблоны пустых и загрузочных состояний.

**Артефакты:** компонентный каталог (в docs), матрица состояний компонентов.

#### Фаза 3 — Ключевые сценарии (Недели 6–8)

- Редизайн `complaint`:
  - визуальные секции, понятная иерархия;
  - прогресс и контекстные подсказки;
  - улучшенные сообщения об ошибках.
- Редизайн `profile`:
  - структурирование блоков;
  - явные CTA и статусные индикаторы.

**Артефакты:** обновлённые шаблоны экранов, UX-отчёт до/после.

#### Фаза 4 — HTMX и полировка (Недели 9–10)

- Добавить HTMX точечно в дорогие по времени взаимодействия.
- Уменьшить количество полных перерисовок страницы.
- Отполировать микро-взаимодействия и анимации (150–200ms).

**Артефакты:** список HTMX endpoint-ов, карта частичных обновлений.

#### Фаза 5 — Стабилизация и релиз (Недели 11–12)

- Регресс по ключевым сценариям (desktop/mobile).
- Accessibility-check (focus order, контраст, видимость фокуса).
- Сравнение KPI до/после и выпуск финального отчёта.

**Артефакты:** релизный чеклист, итоговые KPI, backlog следующих улучшений.

### Definition of Done (для каждой UI-задачи)

Задача считается завершённой, если:

1. Изменение соответствует дизайн-токенам и компонентным правилам.
2. Проверены состояния `default/hover/focus/disabled/loading/error`.
3. Проверена адаптивность минимум на 360/768/1024/1440 px.
4. Не сломан ключевой пользовательский сценарий.
5. Обновлена документация (если добавлен/изменён компонент).
6. QA-проверка пройдена без критических дефектов.

### Чеклист релиза визуальных изменений

- [ ] Нет «локальных» цветов/размеров вне токенов без явного обоснования.
- [ ] Все новые поля форм имеют единый стиль лейблов/подсказок/ошибок.
- [ ] Кнопки и ссылки имеют предсказуемые состояния.
- [ ] Таблицы/карточки корректно схлопываются на мобильных ширинах.
- [ ] Есть пустые/ошибочные/загрузочные состояния для динамических блоков.
- [ ] Метрики зафиксированы и сопоставлены с baseline.

### Карта метрик (что мерить еженедельно)

1. **LCP** (ключевые экраны).
2. **Time-to-task** для сценария `complaint`.
3. **Error rate** валидации по форме.
4. **UI lead time:** время от постановки до релиза UI-правки.
5. **Regression count:** число регрессий на спринт.

### Риски и меры снижения

- **Риск:** несистемные точечные правки ломают консистентность.  
  **Мера:** обязательное ревью на соответствие токенам + DoD.
- **Риск:** затягивание редизайна из-за попытки «переделать всё».  
  **Мера:** фокус на P0/P1, поэкранный rollout.
- **Риск:** деградация производительности из-за визуальных эффектов.  
  **Мера:** лимиты на анимации и контроль LCP в каждом релизе.
- **Риск:** сложность поддержки при росте интерактивности.  
  **Мера:** точечный HTMX вместо ранней миграции в SPA.

## KPI для оценки эффекта

- Время до первого осмысленного рендера (LCP).
- Время выполнения ключевого сценария (заполнение и отправка жалобы).
- Ошибки валидации на пользователя.
- Стоимость изменений: человеко-часы на 1 релиз UI.

## Итог

Если цель — **максимально рациональные затраты**, стартуйте с **SSR-эволюции (A)**,
добавляйте **HTMX точечно (B)** и откладывайте **SPA (C)** до момента, когда
его необходимость подтверждена метриками и продуктовой стратегией.
