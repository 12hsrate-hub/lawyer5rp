# Варианты будущего дизайна и фронтенд-стека (эффективность по затратам)

## Текущая точка проекта

На текущий момент веб-часть уже работает как FastAPI + серверные HTML-шаблоны + отдельные JS/CSS-ресурсы.
Это означает, что базовая SSR-архитектура уже развёрнута и поддерживается командой.

## Что считать «эффективным по затратам»

Для этого проекта эффективность = баланс из 4 факторов:

1. **Скорость вывода улучшений в UI** (time-to-market).
2. **Стоимость разработки и поддержки** (часы команды, сложность кода).
3. **Риск миграции** (вероятность сломать существующие пользовательские сценарии).
4. **Производительность и SEO** (быстрая загрузка страниц, индексируемость).

## Вариант A — Эволюция текущего SSR (рекомендуется как базовый)

### Стек

- FastAPI + Jinja2 (оставляем).
- Текущий vanilla JS, но структурируем по модулям/страницам.
- Токенизированная дизайн-система на CSS-переменных.

### Когда подходит

- Когда нужен быстрый редизайн без полной смены технологической платформы.
- Когда важен бюджет и минимизация рисков.

### Плюсы

- Минимальная стоимость внедрения (используется текущая архитектура).
- Почти нет затрат на инфраструктурную миграцию.
- Хорошо для форм, кабинетов, админок и юридических сценариев с server-first логикой.

### Минусы

- При росте сложных интерактивных экранов JS-слой может усложняться.
- Меньше готовых UI-экосистем, чем у React/Vue.

### Оценка затрат (относительно)

- Вход: **низкий**.
- Поддержка: **низкая/средняя**.
- Риск: **низкий**.

## Вариант B — FastAPI + HTMX (плавный шаг к более «живому» UI)

### Стек

- FastAPI + Jinja2.
- HTMX для частичных обновлений HTML без SPA.
- Минимальный JS для специфичных сценариев.

### Когда подходит

- Когда нужны более «динамичные» формы/таблицы/виджеты,
  но без стоимости полноценного SPA.

### Плюсы

- Очень хорошее соотношение «функциональность/затраты».
- SSR-подход сохраняется, команда не теряет текущие компетенции.
- Упрощает сложные интерактивные участки (пагинация, фильтры, inline-редактирование).

### Минусы

- Потребует внедрения новых UI-паттернов и дисциплины по endpoint-структуре.
- Не заменяет полностью потребность в JS при сложной клиентской логике.

### Оценка затрат (относительно)

- Вход: **низкий/средний**.
- Поддержка: **средняя**.
- Риск: **низкий/средний**.

## Вариант C — SPA на React/Vue + API (максимальная гибкость, выше стоимость)

### Стек

- Отдельный фронтенд (React/Vue + Vite/Next/Nuxt).
- FastAPI как чистый API backend.

### Когда подходит

- Когда продукту нужен сложный интерактивный интерфейс масштаба SaaS.
- Когда есть стабильная фронтенд-команда и бюджет на поддержку двух приложений.

### Плюсы

- Максимальная гибкость UI/UX.
- Широкая экосистема компонентов и библиотек.

### Минусы

- Самая высокая стоимость владения.
- Двойная сборка/деплой/наблюдаемость (frontend + backend).
- Существенные миграционные риски.

### Оценка затрат (относительно)

- Вход: **высокий**.
- Поддержка: **высокая**.
- Риск: **средний/высокий**.

## Рекомендация для lawyer5rp

Оптимальный путь по цене/скорости/риску:

1. **Этап 1 (сейчас): Вариант A** — сделать редизайн в текущей SSR-архитектуре.
2. **Этап 2 (точечно): добавить элементы Варианта B (HTMX)** для самых «живых» экранов.
3. **Этап 3 (опционально): рассматривать SPA только при доказанной бизнес-необходимости**.

Иными словами: **не делать «big bang»-миграцию**.

## Практический план (6–8 недель)

1. **Недели 1–2:** UI-аудит и дизайн-токены.
2. **Недели 2–4:** обновление базового layout, типографики, форм, таблиц, кнопок.
3. **Недели 4–6:** редизайн 2 ключевых страниц (например, `complaint`, `profile`) с UX-метриками.
4. **Недели 6–8:** точечное внедрение HTMX в узких местах (где выигрыш очевиден).

## KPI для оценки эффекта

- Время до первого осмысленного рендера (LCP).
- Время выполнения ключевого сценария (заполнение и отправка жалобы).
- Ошибки валидации на пользователя.
- Стоимость изменений: человеко-часы на 1 релиз UI.

## Итог

Если цель — **максимально рациональные затраты**, стартуйте с **SSR-эволюции (A)**,
добавляйте **HTMX точечно (B)** и откладывайте **SPA (C)** до момента, когда
его необходимость подтверждена метриками и продуктовой стратегией.
