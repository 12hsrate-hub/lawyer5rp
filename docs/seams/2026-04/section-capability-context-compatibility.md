# Compatibility Seam Note

- Seam ID: `seam-2026-04-section-capability-context`
- Area: `web/ogp_web/services/section_capability_context_service.py`
- Reason this seam exists: the first migration slice needs one shared selected-server-aware resolver for `complaint`, `court_claim`, and `law_qa`, but those flows still execute through separate legacy executors and mixed runtime/config paths.
- Current source of truth: selected server still resolves through the existing server context + permission model, while templates/forms/validation/law reads remain split across complaint runtime, document builder metadata, and law QA runtime services.
- Target source of truth: each migrated section should resolve through one shared capability context that eventually reads published server pack truth and selected-server-bound law readiness without route-local fallback branches.
- What changed in this task: `shrunk`
- Why this was necessary: route-local selected-server/config reads for the first migration slice were starting to diverge. The new section capability context keeps the abstraction deliberately thin and makes the current truth, target truth, compatibility bridge, and removal gate explicit per primary section.
- Additional narrowing in the current slice: `complaint`, `court_claim`, and `law_qa` now resolve through one shared selected-server-aware section context for page render, complaint draft/generate paths, document builder bundle resolution, and the law QA submit path. A new runtime read-model endpoint (`/api/runtime/sections/{section}/capability-context`) exposes that context for diagnostics and migration validation.
- Additional narrowing in the latest slice: published artifact resolution for `complaint` and `court_claim` is now also funneled through one shared resolver over runtime pack metadata + published content workflow versions. Complaint keeps using the existing pilot adapter bridge, but it now shares template/form/validation artifact selection with the section context diagnostics; court-claim bundle resolution now publishes the same artifact provenance inside bundle status instead of inferring everything from code-owned defaults alone.
- Additional narrowing in the current slice: `law_qa` now resolves a selected-server-specific law readiness verdict through one shared service before submit and inside runtime diagnostics. The service explicitly distinguishes `projection_bridge`, `legacy_effective_sources`, and `legacy_bundle_fallback` modes, and `law_qa` now fails closed on selected-server runtime drift instead of silently accepting an arbitrary `law_version_id`.
- Additional narrowing in the latest slice: the same shared law-context readiness verdict is now exposed inside `/admin/servers` health/workspace read models. Admin runtime health now reports `law_context` as an explicit check, and the laws/onboarding summaries consume the same `reason_code`, `mode`, and readiness status that runtime `law_qa` uses, instead of maintaining a separate operator-only law readiness heuristic.
- Additional narrowing in the current slice: `/admin/servers` now has an explicit server-pack draft/publish path. Draft and publish-blocker payloads are compiled from the current runtime/config path, published content workspace state, access summary, and the shared selected-server law-context verdict. Publish is now fail-closed when runtime metadata is incomplete or when law-backed readiness is not fully `ready`, so operator flow is starting to converge on `published server pack` as the intended runtime config truth instead of treating packs as a hidden bootstrap artifact.
- Additional narrowing in the latest slice: migrated runtime routes now consume one shared published-runtime requirement verdict. `complaint` generation, `court_claim` bundle fetch, and `law_qa` submit no longer silently invent their own acceptance rules; they all read the same `published_pack` target-truth gate from section capability context diagnostics. Bootstrap-pack compatibility remains allowed for the existing baseline server, but neutral-fallback execution is now explicitly blocked through one shared service instead of being left to route-local behavior.
- Additional narrowing in the current slice: migrated render and execution paths now share one section-access verdict. `court_claim` and `law_qa` page render no longer depend on route-local permission branching alone; page render, bundle fetch, and submit all resolve access from the same section capability context and expose that verdict in runtime diagnostics. This keeps v1 on top of existing RBAC without introducing a new policy DSL.
- Additional narrowing in the latest slice: migrated artifact/config readers now share one runtime-pack reader. `published_artifact_resolution`, `document_builder_bundle_service`, the complaint pilot adapter, and legacy generation context snapshot no longer reach into runtime registry helpers independently for pack metadata. `court_claim` bundle now builds from base schema + overlay resolved from the shared runtime-pack reader, and complaint adapter/generation snapshot resolve pack version and template/validation bindings through the same seam instead of direct `effective_server_pack(...)` reads.
- Additional narrowing in the current slice: `law_context_readiness` is now runtime-pack-aware. Canonical bindings/projection truth still drive readiness, but fallback source URLs and bundle metadata now prefer the shared runtime-pack snapshot and expose pack provenance (`resolution_mode`, `pack_version`) in runtime/admin diagnostics before falling back to legacy server-config law context. This shrinks another hidden config read path without changing the current compatibility modes.
- Additional narrowing in the latest slice: bootstrap-pack compatibility is no longer a global implicit allowance for every migrated flow. The shared published-runtime gate now reads a capability-level compatibility policy and supports staged strict cutover through `OGP_STRICT_PUBLISHED_RUNTIME_SECTIONS`. `law_qa` and later `court_claim` now use that same shared mechanism to reach `published_pack only` runtime truth without new route branching or another ad hoc rollout shim, while `complaint` still remains on a staged compatibility path until its own cutover window.
- Additional narrowing in the current slice: `/admin/servers` now exposes migrated runtime truth using the same shared gate that runtime routes use. Health/workspace payloads include per-capability verdicts for `complaint`, `court_claim`, and `law_qa`, plus a summarized `runtime_requirements` block that distinguishes `ready`, `ready_with_compatibility`, and `blocked`. The server card overview now surfaces that same truth so operators can see which migrated routes are genuinely runtime-ready before a production cutover.
- Additional narrowing in the latest slice: runtime server packs now have an explicit rollback path. Admin APIs can roll the current published pack back to the previous published version (or a specified published version) by creating a new published fact from the older metadata, and the server workspace now exposes whether a rollback target exists. This reduces the remaining “deploy-and-pray” risk around strict runtime cutovers.
- Additional narrowing in the current slice: `law_qa` strict published-runtime cutover is now enabled by default, with an explicit rollback lever. The shared published-runtime gate treats `law_qa` as `published_pack only` unless `OGP_RELAXED_PUBLISHED_RUNTIME_SECTIONS=law_qa` is set, so the capability no longer silently accepts bootstrap-pack compatibility as its steady-state runtime path. Primary page renders (`/complaint`, `/court-claim-test`, `/law-qa-test`) now also pass through the same shared runtime gate used by submit/bundle routes, which removes another class of route-local readiness drift.
- Additional narrowing in the latest slice: `law_qa` page/runtime helper reads are more pack-aware. `pages_runtime_service` now prefers shared law-context readiness and runtime-pack law-source metadata before falling back to legacy server-law helpers, and the `ai_service` mask-exception helper now resolves its law bundle path from runtime-pack metadata before touching legacy server config. This does not redesign the retrieval stack, but it removes another hidden split between strict published-runtime gating and the auxiliary law-aware helpers around it.
- Additional narrowing in the current slice: complaint generation snapshot building is now more runtime-pack-first. The legacy snapshot path still exists for compatibility, but it now resolves law-bundle provenance and feature flags from runtime-pack metadata before falling back to legacy server-config helpers. This keeps generated-document provenance closer to the published runtime envelope and reduces one more hidden dependency on the older server-config bridge.
- Additional narrowing in the latest slice: `/admin/servers` onboarding/readiness now treats explicit published-pack metadata as authoritative for law-source and rollout-default checks. If a published pack explicitly clears `law_qa_sources`, `law_qa_bundle_path`, or `feature_flags`, the admin health/workspace payload no longer treats code-level runtime defaults as proof that the server is workflow-ready or rollout-ready. That shrinks another operator-side compatibility seam between “pack truth” and “runtime_config defaults”.
- Additional narrowing in the current slice: shared published-artifact resolution now also respects explicit empty pack bindings. If a published pack explicitly clears `template_bindings` or `validation_profiles`, the complaint/court-claim artifact resolver no longer silently falls back to code-owned default template or validation keys and instead reports those published artifacts as absent. User-facing bundle contracts stay intact, but diagnostics and artifact provenance no longer pretend that missing published bindings are fully configured.
- Additional narrowing in the latest slice: complaint generation snapshot building now also respects explicit empty pack metadata. If a published pack explicitly clears `feature_flags` or `law_qa_bundle_path`, the legacy snapshot builder no longer pulls old server-config defaults into provenance behind the scenes. Generation contracts stay unchanged, but the snapshot now reflects the published runtime envelope more honestly instead of masking missing pack state with legacy defaults.
- Additional narrowing in the current slice: `complaint` and `court_claim` now use the same staged strict-cutover lever as `law_qa`. Bootstrap-pack compatibility is no longer hard-coded as permanently allowed: the shared runtime gate can move either capability to `published_pack only` without adding route-local rollout logic, and later `court_claim` used that path to become strict-by-default.
- Additional narrowing in the latest slice: server-pack publish blockers now evaluate the shared runtime-gate outcome for the *candidate published pack*, not just the current live resolution path. This means strict cutover for `complaint` or `court_claim` no longer falsely blocks pack publication while the server is still on bootstrap compatibility, and law-backed runtime failures show up as explicit candidate runtime blockers before publish. That shrinks the remaining gap between admin publish validation and runtime truth.
- Additional narrowing in the current slice: `/admin/servers` overview now surfaces both the current migrated runtime truth and the candidate post-publish runtime truth from pack blockers. Operators can now see, in one place, whether the live server is still on a compatibility path and whether the next pack publish would actually move the server to a cleaner runtime state. This reduces another operator-side split between “what runtime is doing now” and “what pack publish would do next.”
- Additional narrowing in the latest slice: candidate publish truth for `court_claim` now requires explicit published artifact bindings, not just generic pack metadata. Server-pack publish blockers and candidate runtime requirements mark `court_claim` as blocked unless the candidate pack explicitly declares `template_bindings.court_claim` and `validation_profiles.court_claim_default`, which brings admin-side publish validation closer to the strict runtime boundary that `court_claim` will use after cutover.
- Additional narrowing in the current slice: `/admin/servers` workspace now surfaces one shared `runtime_focus` read model derived from current runtime truth and candidate publish truth. Instead of making operators infer the next cutover blocker from separate health cards, the workspace now points directly at the first migrated capability that is still blocked or still only compatibility-ready, along with a short next-action hint. This keeps operator workflow closer to the same shared runtime gate family that live routes and pack publish validation already use.
- Additional narrowing in the latest slice: workspace onboarding state is now stricter about runtime truth convergence. The `/admin/servers` overview no longer treats `workflow-ready` and `rollout-ready` as purely inherited metadata states when candidate publish truth is blocked or when publish blockers are still open; instead, the workspace summary downgrades those states until current/candidate migrated runtime truth and pack publishability are aligned. This shrinks another operator-side gap between legacy readiness heuristics and the shared runtime gate.
- Additional narrowing in the current slice: `/admin/servers` now exposes `strict_cutover_candidates` for migrated flows. This read model is derived from the same current/candidate runtime truth and rollout env levers (`OGP_STRICT_PUBLISHED_RUNTIME_SECTIONS`, `OGP_RELAXED_PUBLISHED_RUNTIME_SECTIONS`) that runtime gating already uses, so operators can see whether `court_claim` or `complaint` is still blocked, ready to flip from compatibility mode, already safe because the live path is published-pack-backed, or already strict-active. That removes another piece of guesswork from the controlled cutover workflow without adding a parallel rollout mechanism.
- Additional narrowing in the latest slice: complaint strict-prep now follows the same explicit candidate-truth rule as `court_claim`. Server-pack publish blockers fail closed when the candidate pack does not explicitly declare complaint template/validation bindings, and the strict-cutover operator view now includes a complaint/court-claim-specific `next_action` instead of leaving those missing bindings implicit. This shrinks another hidden path where complaint could appear “almost ready” even though the next published runtime envelope still lacked explicit published artifacts.
- Additional narrowing in the current slice: `court_claim` strict rollout is now regression-covered end-to-end. Tests now verify that `OGP_STRICT_PUBLISHED_RUNTIME_SECTIONS=court_claim` blocks bootstrap-pack page/bundle/runtime paths while leaving published-pack-backed `court_claim` paths operational. This does not flip strict mode by default, but it removes more uncertainty from the actual rollout packet by proving that the shared gate behaves correctly at route level, not only inside diagnostics and unit-level verdicts.
- Additional narrowing in the latest slice: `court_claim` strict cutover is now enabled by default, with rollback through `OGP_RELAXED_PUBLISHED_RUNTIME_SECTIONS=court_claim`. Bootstrap-pack `court_claim` paths no longer count as normal steady-state runtime, so page/bundle/runtime checks now assume a published-pack-backed path unless the capability is explicitly relaxed. This shrinks one of the last major compatibility allowances for migrated non-law flows and turns the earlier rollout packet into the new default posture.
- Additional narrowing in the current slice: `complaint` strict rollout is now regression-covered end-to-end in the same way `court_claim` was covered before its default flip. Tests now verify that `OGP_STRICT_PUBLISHED_RUNTIME_SECTIONS=complaint` blocks bootstrap-pack complaint page/generate paths while leaving published-pack-backed complaint paths operational. This does not flip complaint to strict-by-default yet, but it removes another chunk of uncertainty from the final complaint cutover packet and keeps the rollout mechanism shared instead of introducing complaint-specific branching.
- What is still intentionally transitional:
  - complaint generation still bridges through `pilot_runtime_adapter` / legacy snapshot construction
  - law QA still uses the current retrieval / activation stack rather than a full published law-context pack
  - admin onboarding still keeps legacy `workflow-ready` / `rollout-ready` state semantics; shared law-context readiness is now surfaced there, but it does not yet hard-gate pack publish or onboarding state transitions
  - the new server-pack compiler currently seeds candidate metadata from the current runtime/config path and bounded admin-authored metadata updates; it is not yet a full replacement for all legacy config sources
  - the new published-runtime requirement still treats `bootstrap_pack` as an allowed compatibility mode for `complaint`, so the final convergence to `published_pack` runtime truth is not finished yet
  - section access is now shared for migrated flows, but it still resolves only from existing RBAC/role assignments; server-scoped resource policies are not implemented yet
  - runtime pack reads are now shared for migrated readers, but the global server registry still exposes older pack/fallback helpers for non-migrated paths
- Rollback path: routes can fall back to direct `resolve_user_server_context(...)` / `resolve_user_server_config(...)` calls because the new section resolver is additive and does not replace executor logic or route contracts.
- Removal gate: remove route-local server/config resolution for these flows once published artifact resolution and selected-server-bound law readiness fully replace the remaining compatibility bridges.

- Remaining risks:
  - the capability registry is intentionally seeded and thin, so later phases must keep it from accumulating executor-owned branching
  - the runtime context currently reports current truth and bridges, but it does not yet enforce published artifact resolution for all flows
  - selected-server alignment is now shared for the first slice, but non-migrated routes can still use older server-resolution paths
  - `/admin/servers` now shares the runtime law-context verdict, and server-pack publish consumes it, but runtime readers for migrated flows still need a later cutover to depend only on the newly published pack instead of compatibility-aware config reads
  - pack draft metadata is currently bounded but still manually editable through the API; later phases should move more of that draft surface to structured editor flows instead of metadata-shaped input
  - the shared runtime gate now blocks neutral-fallback execution for migrated submit/bundle paths and can stage bootstrap-pack cutover per capability, but page render and draft-authoring paths are still intentionally looser until access/render consistency is converged in a later phase
  - page render now shares the runtime gate for primary migrated pages, but complaint-test and other non-primary test surfaces still intentionally keep their older compatibility/access posture
  - page render and runtime execution now share a section-access verdict for migrated flows, but complaint test surfaces and non-migrated routes still carry older permission patterns that should be narrowed later instead of being force-fit into the primary capability contract
  - complaint and court-claim migrated readers now share one runtime-pack reader, but complaint still remains on a later “published pack only, no bootstrap compatibility” cutover phase
  - law_qa page/runtime helpers now prefer runtime-pack-backed law metadata, but the deeper retrieval / activation stack still runs through the existing law runtime services and remains a later convergence step
  - complaint generation provenance now prefers runtime-pack bundle/feature metadata, but the legacy snapshot path still exists and document generation itself has not yet been fully replaced by a pack-only runtime implementation
