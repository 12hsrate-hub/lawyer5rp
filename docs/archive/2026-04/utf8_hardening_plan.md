# UTF-8 hardening plan (forms + full stack)

## Цель

Убрать «кракозябры» и сделать так, чтобы любые пользовательские данные (RU/EN, спецсимволы, emoji) стабильно проходили путь:

`браузер -> HTTP -> FastAPI -> БД/файлы -> API/HTML ответ -> браузер`.

## Что уже зафиксировано в коде

1. Формы в шаблонах принудительно отправляют данные в UTF-8 через `accept-charset="UTF-8"`.
2. HTTP-ответы приложения дополнительно нормализуются middleware: для text/json/xml/javascript-контента автоматически проставляется `charset=utf-8` (если не указан).
3. Добавлены smoke-проверки, подтверждающие UTF-8 charset для HTML-страниц и наличие `accept-charset` у форм.

## Масштабные меры (рекомендуемый стандарт проекта)

### 1) Транспортный контракт (HTTP)

- Для JSON API всегда `Content-Type: application/json; charset=utf-8`.
- Для HTML всегда `Content-Type: text/html; charset=utf-8`.
- Для CSV/текстовых выгрузок — явный charset + при необходимости BOM (для Excel-сценариев).

### 2) Контракт фронтенда

- Все `<form>` должны иметь `accept-charset="UTF-8"`.
- Все `fetch`-запросы отправляют JSON/текст в UTF-8.
- Запрещено вручную конвертировать строку в legacy кодировки (cp1251/windows-1251) в клиентском JS.

### 3) Контракт backend parsing

- Никогда не декодировать request body с `errors="ignore"` для бизнес-данных.
- Для парсеров multipart/urlencoded оставить только UTF-8 режим.
- На ошибках декодирования возвращать 400 с диагностикой (не «молча» портить данные).

### 4) База и драйверы

- PostgreSQL: `server_encoding='UTF8'`, `client_encoding='UTF8'`.
- PostgreSQL: хранить текстовые и JSON-поля только в UTF-8, при миграциях валидировать текст.
- Для JSON сериализации: `ensure_ascii=False` (в проекте уже активно используется).

### 5) Файлы и логирование

- Все `read_text/write_text/open` только с явным `encoding="utf-8"`.
- Логи — UTF-8 handler (в проекте уже включён).
- Прогон `scripts/check_utf8.py` в CI как обязательный step.

### 6) CI/тест-контур

- Добавить отдельный e2e-тест: отправка формы с кириллицей/emoji, затем чтение из API и сверка байт-в-байт.
- Добавить property-based тест для набора Unicode символов.
- Проверять headers (`content-type` с charset) для ключевых HTML/API роутов.

### 7) Эксплуатация и мониторинг

- Метрика ошибок декодирования/валидации по endpoint.
- Алерт, если растёт доля 400/422 с причиной encoding/parse error.
- Runbook для саппорта: как локализовать источник порчи (браузер, reverse proxy, backend, DB).

## Быстрый чеклист для новых фич

- [ ] Форма имеет `accept-charset="UTF-8"`.
- [ ] Endpoint и response возвращают корректный `Content-Type` с charset.
- [ ] Тест с кириллицей и спецсимволами добавлен.
- [ ] Нет `errors="ignore"` в пользовательском data-flow.
- [ ] Все новые файлы/миграции/экспорты читаются и пишутся в UTF-8.
