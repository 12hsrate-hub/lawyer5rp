# ADR: Staged Migration Strategy Inside Current Monolith

- Status: Accepted
- Date: 2026-04-14
- Owners: Platform Architecture

## Context

Проект развивается как действующий монолит с высокой связностью между доменными зонами, большим количеством legacy-маршрутов и исторически сложившимися server-правилами в Python-константах. Полный rewrite несёт высокий риск по срокам, совместимости и операционной устойчивости.

Нужна формальная архитектурная позиция, которая:

1. фиксирует стратегию постепенной миграции внутри текущего монолита;
2. замораживает появление новых legacy-паттернов;
3. задаёт целевые bounded contexts для декомпозиции;
4. вводит governance миграций через roadmap-фазы и acceptance criteria.

## Decision

### 1) Baseline strategy: staged migration in the current monolith

Базовой стратегией принимается **staged migration внутри текущего monolith**.

**Full rewrite не является целевой стратегией** и не должен использоваться как baseline-подход для эволюции системы.

Архитектурные изменения выполняются инкрементально, с сохранением работоспособности продакшн-контуров на каждом этапе.

### 2) Legacy pattern freeze

С момента принятия ADR вводится заморозка legacy-паттернов:

- **No new hardcoded server rules в Python-константах.**
  Все новые server-правила должны задаваться через конфигурационные/реестровые механизмы, совместимые с целевой архитектурой.

- **No новые legacy-роуты.**
  Новая функциональность должна реализовываться только в рамках целевых контрактов и контекстов, без добавления новых legacy endpoint'ов.

Существующие legacy-артефакты допускаются только как compatibility layer до соответствующих roadmap-фаз миграции.

### 3) Target bounded contexts

Для декомпозиции монолита фиксируются следующие целевые bounded contexts:

1. **Platform Core**
2. **Admin**
3. **Server Packs**
4. **Law Registry**
5. **Workflow Engine**
6. **AI/Retrieval**
7. **Publishing/Audit**

Любая новая архитектурная инициатива должна быть явно привязана минимум к одному из этих контекстов.

### 4) Migration governance

Вводится правило governance: **все изменения миграции выполняются только через roadmap-фазы и acceptance criteria**.

Обязательные требования:

- каждая инициатива должна ссылаться на конкретную roadmap-фазу;
- каждая фаза должна иметь измеримые acceptance criteria;
- переход к следующей фазе разрешён только после формального подтверждения criteria текущей фазы;
- изменения, не привязанные к roadmap-фазе и criteria, не принимаются.

## Consequences

- Снижается архитектурный риск за счёт отказа от full rewrite как стратегии по умолчанию.
- Останавливается рост технического долга от новых hardcoded server-правил и legacy-роутов.
- Появляется единая карта декомпозиции через bounded contexts.
- Управление миграцией становится проверяемым и воспроизводимым благодаря фазам и acceptance criteria.
