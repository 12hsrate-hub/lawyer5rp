# Multi-server visual admin platform blueprint

> Стартовый blueprint для перехода от «админки как набора страниц» к единой визуальной multi-server платформе управления.

## 1) Цели

- Единая точка администрирования для:
  - runtime-серверов;
  - источников законов и их привязок;
  - контент-воркфлоу и фоновых задач;
  - качества AI (cost / integrity / retry / rollback).
- Предсказуемый UX: каждый сценарий имеет статус, историю изменений и безопасный rollback.
- Изоляция данных per-server: отсутствие скрытых global-связей для runtime-конфигов.

## 2) Текущая опора в коде (что уже можно переиспользовать)

- API и UI для runtime servers и server-law bindings уже присутствуют в админ-модуле.
- Есть персистентные слои для `runtime_servers_store`, `runtime_law_sets_store`, `admin_catalog_store`, `admin_dashboard_repository`.
- Есть слой async jobs и миграции БД для ops-процессов.

Это позволяет начать с эволюции текущей админки, а не с переписывания «с нуля».

## 3) Целевая архитектура (UI-first)

### 3.1 Визуальные домены панели

1. **Servers**
   - список серверов, статус, активность, health;
   - CRUD + soft-disable;
   - техпрофиль (модель, лимиты, URL, фичефлаги).

2. **Law Sources**
   - реестр источников;
   - связь источник ↔ сервер (many-to-many);
   - запуск rebuild/refresh с прогрессом.

3. **Quality & Rollout**
   - метрики factual integrity / retry / cost uplift;
   - ручное и авто-управление rollout этапами;
   - аварийный rollback (`force_mode=factual_only`, `emergency_off=true`).

4. **Operations**
   - очередь async jobs, фильтры, перезапуск/отмена;
   - аудит действий администраторов;
   - инцидентные runbook-подсказки в интерфейсе.

### 3.2 Контур безопасности

- RBAC на уровне операций и server scope.
- Явная проверка прав на cross-server действия.
- Все мутации через audit trail (кто, когда, что изменил, old/new).

### 3.3 Контур надежности

- Идемпотентные job-операции для rebuild/import.
- Двухфазные изменения критичных конфигов:
  1) validate/dry-run,
  2) apply.
- Автопереключение в safe-mode при нарушении quality gates.

## 4) План внедрения по этапам

## Phase A — Stabilize foundation (1 спринт)

- Зафиксировать единый контракт payload/response для admin API.
- Довести консистентность ошибок/валидации в runtime servers + law sets + jobs.
- Добавить единый frontend state-модуль для админ-панели (без смены фреймворка).

**Definition of done:**
- все admin endpoints имеют единообразные схемы ответов;
- ключевые пользовательские сценарии покрыты smoke-тестами.

## Phase B — Visual workflows (1–2 спринта)

- Добавить мастер-потоки:
  - «Создать сервер → привязать законы → проверить health → активировать»;
  - «Обновить источник закона → rebuild → валидация качества → publish».
- Внедрить прогресс-виджеты async jobs и журнал операций.

**Definition of done:**
- сценарии выполняются без ручных SQL/CLI шагов;
- есть визуальный статус каждого шага и причина ошибок.

## Phase C — Quality command center (1 спринт)

- Собрать единый экран quality/cost/retry по серверам.
- Добавить кнопки guarded-действий rollout/rollback.
- Визуализировать SLO/гейты и предупреждения до деградации.

**Definition of done:**
- оператор может принять решение о rollout/rollback из UI;
- изменения логируются и воспроизводимы через audit trail.

## Phase D — Multi-server scale-out (дальше по приоритету)

- Шаблоны конфигов для быстрого онбординга нового сервера.
- Массовые операции с защитой (batch change + preview impact).
- Per-server SLA панели и алерты.

## 5) Ближайший практический старт (что делаем первым)

1. Подготовить «инвентаризацию» существующих admin endpoint-ов и привести их к единому формату ошибок.
2. Вынести в `web/ogp_web/static/pages/admin.js` унифицированный модуль состояния/уведомлений для разделов runtime servers, law sets и jobs.
3. Добавить API/UI для просмотра истории действий админов (read-only версия как первый шаг).
4. Закрыть это набором тестов API + web pages + snapshot сценариев.

## 6) KPI на первую итерацию

- < 5 минут на запуск нового сервера через UI (без SQL).
- 100% критичных admin-операций имеют запись в audit log.
- 0 «немых» ошибок: каждое падение возвращает actionable сообщение.
- Снижение ручных действий в деплое/обновлении законов минимум на 50%.

## 7) Риски и как страхуемся

- **Риск:** рост сложности `admin.js`.
  - **Контрмера:** модульная декомпозиция по доменам + контрактные тесты.
- **Риск:** race conditions в rebuild/job flows.
  - **Контрмера:** idempotency keys и optimistic locking для статусов.
- **Риск:** случайные cross-server изменения.
  - **Контрмера:** server-scope guards + подтверждение в UI + RBAC.

## 8) Правила выполнения

- Не ломать текущие API-контракты без слоя совместимости.
- Любое изменение rollout-механики сопровождается обновлением runbook.
- Каждый этап закрывается демонстрацией полного пользовательского сценария «от клика до результата».
