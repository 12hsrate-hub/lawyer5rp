# Redesign plan: admin panel (FastAPI + Jinja2 + vanilla JS)

## 1) Краткая оценка текущего состояния админки

Текущая админка уже функциональна: есть обзор, фильтры, таблица пользователей, события, экспорт CSV, модальное окно действий по пользователю и блок производительности. Всё это собрано на одном SSR-экране `admin.html` + одном JS-файле `admin.js`. Однако продуктово это пока «технический монитор + ручные действия», а не удобная операционная консоль для нетехнических сотрудников.

Что уже хорошо:
- Есть базовый контроль доступа (доступ только для администратора). 
- Есть операции управления аккаунтом (блок/разблок, верификация email, роли tester/gka, смена email, сброс пароля).
- Есть операционные метрики и audit trail через `metric_events`.

Что не дотягивает до зрелой продуктовой админки:
- Одна страница перегружена секциями и терминами.
- Сильный технический уклон формулировок и метрик.
- Нет явной сущности «роли и права» как отдельного сценария.
- Опасные действия недостаточно защищены UX-подтверждениями.

## 2) Главные проблемы текущей админки

1. **Перегрузка одной страницы**: фильтры, сводка, performance, импорт, пользователи, аудит, события — всё на одном полотне. Это увеличивает когнитивную нагрузку.
2. **Смешение слоёв данных**: пользовательские KPI, технические API-метрики и AI/импорт-метрики визуально смешаны.
3. **Недостаточная иерархия действий**: частые действия и рискованные действия выглядят почти одинаково.
4. **Ролевой сценарий неполный**: роли реализованы как флаги (`is_tester`, `is_gka`), но UX не объясняет «что даёт роль», «какие есть ограничения», «что изменится после выдачи».
5. **Непрозрачность времени и контекста метрик**: много накопительных чисел без нормального периода/сравнения.
6. **Фрагментированная читаемость событий**: события есть, но нет удобной типологии (бизнес/безопасность/ошибки/инфраструктура).

## 3) Что сейчас мешает обычному человеку нормально ей пользоваться

- Непонятные подписи (например, `AI OCR`, `RU`, `p95/p50`, `endpoint`).
- Много чисел без ответа на вопрос «и что мне с этим делать?».
- Выдача ролей через одинаковые кнопки без явного предупреждения о последствиях.
- Для ряда действий используются `prompt`, что неудобно и небезопасно для операционной рутины.
- Нет «guided flow» для типовых задач: «найти пользователя → проверить статус → выдать роль → подтвердить и зафиксировать причину».

## 4) Какие метрики должны быть переведены в более человеческий вид

Ниже предложен «человекочитаемый словарь метрик».

- `users_total` → **«Всего аккаунтов»** (зачем: понимать охват системы).
- `events_last_24h` → **«Активность за сутки»** (зачем: видеть, жива ли система операционно).
- `complaints_total` → **«Подготовлено жалоб»** (зачем: бизнес-объём).
- `rehabs_total` → **«Подготовлено заявлений на реабилитацию»** (зачем: объём по второму ключевому сценарию).
- `api_requests_total` → **«Системных операций обработано»** (зачем: нагрузка, но без техно-жаргона).
- `avg_api_duration_ms` → **«Среднее время ответа системы»** (зачем: UX-скорость).
- `error_rate`/ошибки → **«Ошибок в операциях»** + доля (%), с цветовым статусом.
- `pending_scores` → **«Ожидают проверки экзамена»** (зачем: очередь ручного внимания).
- `ai_exam_failure_total` → **«Сбоев при проверке экзаменов»** (зачем: операционный риск).
- `resource_units_total` → **«Расход системных ресурсов»** (и tooltip «внутренняя метрика нагрузки»).

Для каждой метрики в UI нужны:
- **основное число**,
- **период** (сейчас / 24ч / 7д / 30д),
- **изменение к прошлому периоду**,
- **ссылка на drill-down** (таблица/детализация).

## 5) Как должна выглядеть новая структура админки

Предлагаемая IA (без SPA, в рамках Jinja2):

1. `/admin` — **Dashboard (операционный центр)**.
2. `/admin/users` — **Пользователи** (поиск, фильтры, массовые действия low-risk).
3. `/admin/users/{username}` — **Карточка пользователя** (статус, роли, история, действия).
4. `/admin/roles` — **Роли и права** (матрица ролей, назначения, журнал изменений ролей).
5. `/admin/stats` — **Статистика** (бизнес- и операционные срезы, период/сравнение).
6. `/admin/audit` — **Журнал действий** (admin actions + security + errors).
7. `/admin/system` — **Системные статусы** (производительность, ошибки, импорт экзаменов).

Это разделяет сценарии по целям и снижает перегрузку одной страницы.

## 6) Как должен выглядеть dashboard

Dashboard должен отвечать на 5 вопросов за 10–20 секунд:
1. Что происходит сейчас?
2. Где проблемы?
3. Что требует действия администратора?
4. Что изменилось относительно прошлого периода?
5. Куда перейти для решения?

Рекомендуемые блоки:
- **Ключевые KPI (6–8 карточек)**: Аккаунты, Активность за сутки, Жалобы, Реабилитации, Ошибки операций, Очередь экзамена, Заблокированные аккаунты, Неподтверждённые email.
- **Проблемы/алерты**: «Высокая доля ошибок», «Рост блокировок», «Очередь > порога», «Сбой импорта».
- **Лента важных событий** (последние 10): только приоритетные типы.
- **Быстрые переходы**: «Пользователи с блокировкой», «Неподтверждённые email», «Ошибки за 24ч».

Что убрать с dashboard глубже:
- Детальные таблицы top endpoints, длинные событийные таблицы, технические breakdown'ы AI-пайплайна.

## 7) Как должна выглядеть статистика

Раздел `/admin/stats` с фильтрами:
- Период: сегодня / 7 дней / 30 дней / custom.
- Срезы: по ролям, статусу аккаунта, типу события, источнику (если релевантно).

Композиция:
- **Summary cards** (верх): 6–8 ключевых агрегатов.
- **Тренды** (линейные графики только для 2–4 метрик): активность, ошибки, генерации документов.
- **Таблицы breakdown**: по типам событий, по ролям пользователей, по статусам.

Принцип: график только там, где нужен тренд. Для состава/распределения — таблица с процентами лучше круговых диаграмм.

## 8) Как должна выглядеть выдача ролей и управление доступами

### Целевой UX выдачи роли
1. Открыть карточку пользователя.
2. Увидеть блок **«Текущие роли»** + **«Что дают роли»**.
3. Выбрать изменение роли (toggle/кнопка).
4. Получить **диалог подтверждения**:
   - что меняется,
   - какие права добавляются/снимаются,
   - поле «Причина изменения» (обязательно для risk-действий).
5. Сохранить и увидеть запись в **истории изменений ролей**.

### Правила безопасности UX
- Разделить кнопки на: **частые** (нейтральные) и **опасные** (danger zone).
- Для блокировки/сброса пароля/снятия критичных ролей — двухшаговое подтверждение.
- Для email/password — отдельные формы в модальном диалоге вместо `prompt`.
- Отображать **effective permissions** простым языком: «может импортировать экзамены», «может открывать тестовые страницы», «доступ к админке: да/нет».

## 9) Какие страницы и сценарии должны быть приоритетными

1. **Dashboard** (ежедневный вход, triage).
2. **Users list** (поиск + быстрые фильтры + статусные бейджи).
3. **User details** (основной центр действий по одному пользователю).
4. **Roles & permissions** (матрица ролей + журнал изменения ролей).
5. **Audit log** (контроль действий админов и расследование инцидентов).

## 10) Что нужно изменить в backend-подготовке данных

Нужен admin-specific слой представления данных (view-model), а не отдача «сырых» агрегатов напрямую.

### Нужные view-model структуры
- `AdminDashboardVM`
  - `kpis[]` (id, label, value, period, delta, trend, status).
  - `alerts[]` (severity, title, description, action_url).
  - `quick_links[]`.
- `AdminUserListItemVM`
  - статусные поля уже предрасчитаны (human labels, risk flags, last activity label).
- `AdminUserDetailsVM`
  - roles, effective_permissions, security_state, recent_admin_actions.
- `AdminRoleChangeVM`
  - before/after, actor, reason, created_at.

### Нужные endpoint/агрегации
- `GET /api/admin/dashboard` — только верхнеуровневые KPI + alerts.
- `GET /api/admin/users` — пагинируемый список + фильтры/сортировки.
- `GET /api/admin/users/{username}` — детальный VM.
- `POST /api/admin/users/{username}/roles` — атомарное изменение ролей c reason.
- `GET /api/admin/role-history` — история ролевых изменений.
- `GET /api/admin/stats` — агрегаты по периоду и срезам.

### Почему это нужно
Сейчас `/api/admin/overview` перегружен и тянет сразу слишком много разнотипных данных, что мешает масштабированию интерфейса и кэшированию по смысловым зонам.

## 11) Что нужно изменить в Jinja2-шаблонах

- Разбить `admin.html` на модульные partials:
  - `partials/admin/dashboard_kpis.html`
  - `partials/admin/alerts.html`
  - `partials/admin/users_table.html`
  - `partials/admin/user_actions_modal.html`
  - `partials/admin/audit_table.html`
- Ввести единый шаблон каркаса админки (`layouts/admin_base.html`) с левой навигацией разделов.
- Добавить reusable-макросы:
  - карточка метрики,
  - статусный бейдж,
  - пустое состояние,
  - ошибка/предупреждение,
  - строка таблицы с secondary text.

## 12) Что нужно изменить в CSS-архитектуре

Сейчас админ-стили находятся в большом `legal.css`. Для масштабирования:
- Выделить `static/styles/admin/`:
  - `admin.tokens.css` (цвета, отступы, размеры, статусы),
  - `admin.layout.css`,
  - `admin.components.css`,
  - `admin.pages.css`.
- Сохранить текущую дизайн-систему, но отделить админ-слой от публичных/legal-страниц.
- Ввести единые utility-классы состояния: `is-warning`, `is-danger`, `is-empty`, `is-loading`.

## 13) Что нужно изменить в vanilla JS

- Разбить текущий `admin.js` по зонам ответственности:
  - `admin-api.js` (запросы и сериализация фильтров),
  - `admin-state.js` (state + live refresh),
  - `admin-renderers.js` (отрисовка блоков),
  - `admin-actions.js` (операции по пользователю/ролям),
  - `admin-init.js` (инициализация обработчиков).
- Перейти с `window.prompt` на модальные формы с валидацией.
- Добавить централизованный слой `confirmDangerAction(config)`.
- Добавить URL-state синхронизацию фильтров (query params), чтобы ссылки на фильтрованные экраны были воспроизводимы.

## 14) Поэтапный план внедрения без сроков

1. **Стабилизировать данные и контракты (backend foundation)**
   - Ввести VM-структуры и новые endpoint-агрегации.
   - Зафиксировать словарь метрик и единую таксономию событий.
2. **Информационная архитектура и шаблонный каркас**
   - Сделать `admin_base` и разделить админку на несколько страниц.
   - Перенести перегруженные секции из dashboard в специализированные разделы.
3. **Ролевой и пользовательский сценарий**
   - Страница пользователя + безопасные формы действий.
   - Отдельный раздел ролей/прав и история изменений ролей.
4. **Статистика и аудит**
   - Вынести статистику в отдельный раздел с периодами и срезами.
   - Вынести аудит в отдельную таблицу с приоритетной фильтрацией.
5. **CSS/JS систематизация**
   - Модульные CSS и JS файлы, единые компоненты состояния.
6. **Полировка UX**
   - Пустые состояния, helper-тексты, tooltips, предупреждения, доступность.

## 15) Что даст максимальный эффект первыми

1. Разделение на страницы (`dashboard/users/stats/audit/system`) вместо одной «супер-страницы».
2. Переименование метрик и привязка к периодам/дельтам.
3. Безопасный сценарий ролей и опасных действий (без `prompt`, с reason + confirm).
4. Фокус dashboard на «проблемах и действиях», а не на сырых технических числах.

## 16) Какие риски есть у такой переделки

- Риск «косметического» редизайна без backend-подготовки (получится красиво, но всё ещё неудобно).
- Риск нарушения обратной совместимости API для текущего `admin.js`.
- Риск несогласованной терминологии между UI и backend event types.
- Риск потери доверия администраторов, если убрать привычные данные без замены на понятные аналоги.

Снижение рисков:
- Вводить новые endpoint параллельно старым.
- Поддержать режим feature-flag для новых экранов.
- Сначала пилот на `dashboard + user details`, затем остальные разделы.

## 17) Практический вывод: с чего начать прямо сейчас

1. Зафиксировать **словарь админ-метрик в человекочитаемых терминах** и утвердить 8–10 KPI для dashboard.
2. В backend добавить `GET /api/admin/dashboard` и `GET /api/admin/users` (пагинация + фильтры + сортировки).
3. Разделить `admin.html` на `dashboard` и `users` как первые два экрана.
4. Заменить prompt-действия в карточке пользователя на модальные подтверждаемые формы.
5. Добавить журнал изменений ролей как отдельный API и блок на странице пользователя.

После этого уже можно последовательно довести статистику, аудит и системные статусы до зрелого уровня без rewrite стека.
